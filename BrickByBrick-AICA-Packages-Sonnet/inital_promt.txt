Detaillierte Systemarchitektur & Block-Logik
1. Block: ExplorationNavigator
Zweck: Steuert die Exploration, führt den Handshake mit der Kamera und agiert später als passives Gateway.
Initialisierung:
YAML Einlesen: Der Block greift auf den konfigurierten Dateipfad (z. B. ExplCoords.yaml) zu.
Er durchläuft die Datei und wandelt die dort hinterlegten Positionen und Orientierungen in echte AICA CartesianPose-Objekte um.
Diese Posen werden in die interne Variable exploration_pose_list geladen.
Der interne state wird explizit auf "EXPLORATION" gesetzt.
Die Variablen waiting_for_camera_reset und trajectory_success werden auf False gezwungen.

Inputs: ist_pose (cartesian_pose), target_pose_in (cartesian_pose, vom PPL), take_img_in (bool, vom PPL), img_taken (bool, von Kamera)
Outputs: target_pose_out (cartesian_pose), take_img_out (bool), trajectory_success (bool), trigger_ppl (bool)
Interne Variablen: state ("EXPLORATION" oder "GATEWAY"), exploration_pose_list (Array von Posen), waiting_for_camera_reset (bool)
Detaillierte interne Logik (on_step_callback):
Zustandsprüfung: Abfrage, ob state == "EXPLORATION".
Wenn EXPLORATION:
Abfrage, ob Länge der exploration_pose_list > 0 ist.
Wenn JA (Liste nicht leer):
Sende obersten Eintrag (exploration_pose_list[0]) konstant an Output target_pose_out.
Distanzberechnung: Berechne euklidische Distanz (X, Y, Z) zwischen Input ist_pose und target_pose_out.
Zielerreichung: Abfrage, ob Distanz < 0.002 (2mm) UND waiting_for_camera_reset == False.
Wenn JA: Setze trajectory_success = True und take_img_out = True.
Handshake-Warten: Abfrage, ob Input img_taken == True.
Wenn JA: Kamera hat fotografiert. Setze take_img_out = False, trajectory_success = False. Entferne (pop) den Eintrag [0] aus der Liste. Setze waiting_for_camera_reset = True (Sicherheits-Lock).
Sicherheits-Lock lösen: Abfrage, ob waiting_for_camera_reset == True UND img_taken == False.
Wenn JA: Kamera hat das fallende Signal bestätigt. Setze waiting_for_camera_reset = False. Der Zyklus für die nächste Pose beginnt.
Wenn NEIN (Liste leer):
Setze nach 0.5s trigger_ppl = True. Wechsle internen state auf "GATEWAY".
Wenn GATEWAY:
Leite Input target_pose_in stumpf 1:1 auf Output target_pose_out weiter.
Leite Input take_img_in stumpf 1:1 auf Output take_img_out weiter.
Berechne weiterhin die Distanz zwischen ist_pose und target_pose_in. Wenn < 0.002, setze trajectory_success = True, sonst False.

2. Block: PoseTriggeredCamera
Zweck: Schießt verzögerte Snapshots und friert Roboterdaten 100% synchron ein.
Initialisierung: is_delaying = False, img_taken = False
Inputs: take_img (bool), trajectory_success (bool), ist_pose_in (cartesian_pose), cam_ist_pose_in (cartesian_pose, fest im URDF), image_stream (other/Image)
Outputs: img_taken (bool), ist_pose_out (cartesian_pose), cam_ist_pose_out (cartesian_pose), image_out (other/Image)
Interne Variablen: timer_start (Zeitstempel), is_delaying (bool)
Detaillierte interne Logik (on_step_callback):
Trigger-Abfrage: Prüfe ob take_img == True UND trajectory_success == True UND img_taken == False.
Wenn JA: * Prüfe ob is_delaying == False. Wenn ja: Setze is_delaying = True, speichere aktuelle Uhrzeit in timer_start.
Berechne Delta: aktuell - timer_start.
Abfrage, ob Delta >= 0.3 Sekunden.
Wenn JA: (Snapshot-Moment!) Führe harten Deep-Copy aus: Speichere image_stream auf image_out. Speichere ist_pose_in auf ist_pose_out. Speichere cam_ist_pose_in auf cam_ist_pose_out.
Setze Output img_taken = True. Setze is_delaying = False.
Abbruch-Sicherheit: Falls während is_delaying == True das Signal trajectory_success auf False abfällt (Roboter wurde weggeschoben), setze is_delaying = False (Timer bricht ab).
Handshake-Reset: Abfrage, ob trajectory_success == False UND img_taken == True.
Wenn JA: Der Roboter ist abgefahren. Setze img_taken = False.

3. Block: YoloObjectDetector
Zweck: Findet Klötze im Bild, generiert reine 2D-Eckpunkt-Koordinaten und leitet die exakten Aufnahme-Posen synchron an den DataHandler weiter.
Initialisierung: Während der on_setup-Phase lädt dieser Block das trainierte YOLOv11-OBB Modell (die .pt-Gewichtsdatei) von der Festplatte in den Arbeitsspeicher (RAM/GPU). Dies ist ein zeitintensiver Vorgang, der zwingend vor dem ersten Bild passieren muss, damit der laufende Betrieb später nicht blockiert wird.
Der Output yolo_done_trigger wird auf False gesetzt. Der Block ist nun passiv und wartet auf das erste Bild-Event.

Inputs: image_in (other/Image, Event-Trigger), ist_pose_in (cartesian_pose), cam_ist_pose_in (cartesian_pose)
Outputs: yolo_corners_list (double_array), ist_pose_out (cartesian_pose), cam_ist_pose_out (cartesian_pose), yolo_done_trigger (bool)
Detaillierte interne Logik (user_callback auf image_in):
Event gefeuert: Das eingefrorene Bild (samt Posen) kommt an.
Jage das Bild durch das trainierte YOLOv11-nano Modell.
Platzhalter für das vorbereitete Modell(Pfad)
Extrahiere die gefundenen Masken. 
Masken, die den Rand berühren (näher als 5 Pixel am Rand sind) werden aussortiert.
Lege für die übrigen Masken Tangenten an, um exakt 4 Eckpunkte (P1, P2, P3, P4) im 2D-Pixelraum (u, v) zu bestimmen.
Schreibe alle Eckpunkte hintereinander in ein flaches 1D-Array (Syntax: [u1, v1, u2, v2, u3, v3, u4, v4, u5, v5...]).
Platzhalter für die Eckpunktberechnung und Orientierungsbestimmung
Sende dieses Array an den Output yolo_corners_list.
Posen-Routing: Leite den Input ist_pose_in stumpf 1:1 auf den Output ist_pose_out weiter. Leite den Input cam_ist_pose_in stumpf 1:1 auf den Output cam_ist_pose_out weiter.
Setze yolo_done_trigger = True. (Im regulären on_step_callback wird dieses Flag im darauffolgenden Takt direkt wieder auf False gesetzt, um eine saubere Flanke für den Linien-Block zu erzeugen).

Zweck: Erkennt Ablagelinien und generiert direkte 3D-TCP-Ablageposen.
Inputs: image_in (other/Image), cam_ist_pose (cartesian_pose), yolo_done_trigger (bool, Event-Trigger), trigger_ppl (bool)
Outputs: line_ex_list (double_array)
Detaillierte interne Logik (user_callback auf yolo_done_trigger steigende Flanke):
Bypass-Prüfung: Abfrage, ob trigger_ppl == True. Wenn JA -> sofortiger Abbruch (Return). Spart CPU in Phase 2.
Lade image_in.
Platzhalter für Bernhard's Skript

5. Block: MasterListManager (DataHandler)
Zweck: Geometrische Berechnungen (Pixel $\rightarrow$ 3D), Verdreh-Optimierung für den Roboterarm, Filterung und Master-Plan Erstellung.
Initialisierung: Die drei zentralen Speicher-Arrays master_overview_out, master_dropoff_out und filtered_yolo_out werden als leere Listen [] initialisiert.
Die Hardware-Konstanten (wie Kameramatrix $f_x, f_y, c_x, c_y$ und die Tisch-Höhe $Z_{Tisch}$) werden als interne Variablen bereitgestellt.
Der Block verharrt passiv, bis das erste Event von YOLO oder dem Linien-Block eintrifft.

Inputs: yolo_corners_list_in (double_array), line_ex_list_in (double_array), ist_pose_in (cartesian_pose, vom YOLO durchgereicht), cam_ist_pose_in (cartesian_pose, vom YOLO durchgereicht), trigger_ppl (bool)
Outputs: master_overview (double_array), master_dropoff (double_array), filtered_yolo (double_array)
Detaillierte interne Logik:
Event: YOLO-Daten kommen an:
Abfrage, ob Länge yolo_corners_list_in > 0.
Wenn JA & trigger_ppl = False: Hänge die aktuelle ist_pose_in (Syntax: X,Y,Z, Qx,Qy,Qz,Qw) als neuen 7-stelligen Block an das Array master_overview an.
Geometrie-Rechnung (Eckpunkte $\rightarrow$ 3D Pose mit Twist-Minimierung):
Wandle die ist_pose_in in Euler-Winkel um und extrahiere den aktuellen Yaw-Winkel (Z-Rotation) des Roboter-TCPs. Speichere diesen als current_robot_yaw.
Nimm in einer Schleife immer 8 Werte (P1 bis P4) aus der yolo_corners_list_in.
Berechne den Diagonalschnittpunkt der 4 Ecken: $u_{center} = \frac{(u_1 + u_3)}{2}$, $v_{center} = \frac{(v_1 + v_3)}{2}$.
Berechne die Polygonfläche (Gaußsche Trapezformel aus den 4 Ecken) als Maß für die Klotzgröße.
Berechne den Kantenwinkel (die Ausrichtung des Klotzes im 2D-Bild): $\theta = \arctan2((v_2 - v_1), (u_2 - u_1))$.
Platzhalter um diese Berechnung zu implementieren
Projektion (Pinhole): * Wandle Pixel $(u_{center}, v_{center})$ mit Kameramatrix ($f_x, f_y, c_x, c_y$) in einen 3D-Sichtstrahl um.
Rotiere und verschiebe diesen Strahl mit der Input-Variable cam_ist_pose_in ins Weltkoordinatensystem.
Schneide den Strahl mit der Tisch-Ebene (Hardware-Parameter $Z_{Tisch}$). Das ergibt die echten Weltkoordinaten $X_{klotz}, Y_{klotz}$.
Platzhalter Tischhöhe
Verdreh-Optimierung (Twist Minimization): * Ein Klotz ist rechteckig/symmetrisch. Ihn bei Winkel $\theta$ zu greifen, ist mechanisch identisch dazu, ihn bei $\theta + 180^\circ$ zu greifen.
Abfrage: Welcher der beiden Winkel ($\theta$ oder $\theta + 180^\circ$) hat die geringere mathematische Differenz zum current_robot_yaw?
Wähle den Winkel mit der geringsten Differenz und speichere ihn als opt_yaw. (Dadurch verwindet sich der Arm beim Greifen minimal).
Quaternion-Erstellung: Generiere ein 3D-Quaternion, bei dem die Z-Achse/der Sauger zwingend strikt nach unten gezwungen wird (z.B. Roll = $180^\circ$, Pitch = $0^\circ$) UND der Winkel opt_yaw für die Yaw-Rotation angewendet wird.
Zwischenspeichern des fertigen Klotzes im Format [X, Y, Area, Qx, Qy, Qz, Qw]. Die Rotationsachse des Quaternions immer = der negativen World Z-Achse.
Event: Linien-Daten kommen an:
Abfrage, ob Länge(line_ex_list) > Länge(master_dropoff).
Wenn JA: Überschreibe master_dropoff komplett mit der line_ex_list.
Filter-Schleife (Kollisionsprüfung):
(Dieser Teil bleibt unverändert) Iteriere über alle berechneten $(X_{klotz}, Y_{klotz})$.
Berechne euklidische Distanz zu jedem Platz im master_dropoff.
Abfrage: Ist Distanz < 0.01m?
Wenn NEIN: Schreibe Klotz final in den Output filtered_yolo (Syntax: [X, Y, Area, Qx, Qy, Qz, Qw]).
Wenn JA: Klotz liegt bereits auf einer Ablagelinie und wird verworfen (nicht in den Output kopiert).



6. Block: PickPlaceController (PPL)
Zweck: State-Machine für die Roboter-Ausführung in Phase 2.
Initialisierung: Der interne Zustand der State Machine wird strikt auf state = "INIT" gesetzt.
Der Output vacuum_on wird auf False gesetzt, um sicherzustellen, dass der Greifer beim Systemstart deaktiviert ist.
In seiner on_step_callback-Schleife fragt der Block ab jetzt bei jedem Taktzyklus ab: Ist der Input trigger_ppl == True?
Solange dieser Input False ist (was während der gesamten Explorationsphase der Fall ist), überspringt der Block alle weiteren Berechnungen und verharrt im sicheren Standby.
Listen: master_overview_in und master_dropoff kopieren in interne Listen: master_overview_local und master_dropoff_local, damit diese veränderbar sind. Dieses Kopieren darf im Code nur ein einziges Mal passieren! Nämlich exakt in dem Moment, wenn trigger_ppl auf True springt und der Zustand von INIT auf CHECK_LISTS wechselt.

Inputs: trigger_ppl, trajectory_success, img_taken, filtered_yolo (double_array), master_dropoff, master_overview, depth_image (other/Image)
Outputs: target_pose_out, take_img_out, vacuum_on
Zustände: CHECK_LISTS, MOVE_OVERVIEW, WAIT_IMG_1, MOVE_PICK_HOVER, WAIT_IMG_2, EXECUTE_PICK, PREPARE_PLACE, EXECUTE_PLACE
Detaillierte interne Logik (on_step_callback):
CHECK_LISTS:
Abfrage: len(master_overview_local) > 0 UND len(master_dropoff_local) > 0?
Wenn JA -> Wechsle zu MOVE_OVERVIEW.
Wenn NEIN -> target_pose_out = Hardcodierte Home-Pose, Ende.
MOVE_OVERVIEW:
Lies den ersten Werte aus master_overview_local. Schreibe als CartesianPose auf target_pose_out.
Flankenauswertung: Wenn trajectory_success von False auf True springt -> take_img_out = True, wechsle zu WAIT_IMG_1.
WAIT_IMG_1 (Grob-Auswertung & Tiefe):
Wenn img_taken == True: Setze take_img_out = False.
Abfrage: len(filtered_yolo) == 0?
Wenn JA -> Lösche (pop) den obersten Werten aus master_overview_local, wechsle zu CHECK_LISTS.
Wenn NEIN -> Nimm den Klotz mit der höchsten Area aus dem Array.
Tiefen-Berechnung: * Wandle depth_image via image.get_data() in ein Numpy-Array um.
Hole den Median eines 5x5 Patches an der Pixel-Stelle $(u_{center}, v_{center})$ des größten Klotzes. Das ergibt den Tiefenabstand $d$.
Projektion mit Tiefe: Wandle $(u_{center}, v_{center}, d)$ mit Intrinsics in 3D-Kamera-Koordinaten um. Rotiere diesen Punkt mit der aktuellen Kamerapose ins Welt-System -> Das ergibt die echte Klotzhöhe $Z_{pick}$. Da WAIT_IMG_1 und WAIT_IMG_2 unterschiedliche Taktzyklen (States) sind, musst du im Python-Code in WAIT_IMG_1 den Wert zwingend mit self.z_pick_gespeichert = Z_pick in die Klasse schreiben, damit er in WAIT_IMG_2 noch existiert. Lokale Variablen werden am Ende eines Taktzyklus sonst gelöscht.
Hierfür braucht es einen Platzhalter, Format, Berechnung, Übertragung in Weltkoordinaten sind noch nicht vollständig geklärt.
Pose berechnen: Erstelle Hover-Pose $(X_{klotz}, Y_{klotz}, Z_{pick} + hover)$. Die Orientierung ist das berechnete Klotz-Quaternion.
Entscheidung TF-Offset: Da die Kamera einen physischen Abstand zum TCP hat, MÜSSEN wir von $X$ und $Y$ den KameraTCPAbstand aufaddieren, damit die Kamera (und nicht der Sauger!) über dem Klotz zentriert wird.
Platzhalter: Die Umrechnung um die Kamera, statt den Sauger über ein Objekt zu fahren
Wechsle zu MOVE_PICK_HOVER.
MOVE_PICK_HOVER & WAIT_IMG_2 (Fein-Anfahrt):
Sende Hover-Pose. Bei trajectory_success Flanke -> Snapshot triggern.
In WAIT_IMG_2: Verarbeite das frische, zentrierte Bild analog. Nimm größte Maske.
Extrahiere das neue, präzise $(X_{fein}, Y_{fein}, Qx, Qy, Qz, Qw)$.
Generiere Pick-Pose $(X_{fein}, Y_{fein}, Z_{pick} - z\_sauger)$ und Retract-Pose $(X_{fein}, Y_{fein}, Z_{pick} + hover)$.
Wechsle zu EXECUTE_PICK.
EXECUTE_PICK:
Sende Pick-Pose. Bei Ankunft (trajectory_success Flanke) -> Setze vacuum_on = True.
Starte internen Timer (0.3s). Bei Ablauf: Sende Retract-Pose. Bei Ankunft -> Wechsle zu PREPARE_PLACE.
PREPARE_PLACE & EXECUTE_PLACE:
Lies den ersten Wert (X,Y,Z, Qx,Qy,Qz,Qw) aus master_dropoff_local. Die Rotation aus der Linie passt direkt für den TCP!
Sende Hover-Pose ($Z + hover$). Bei Ankunft -> Sende Drop-Pose ($Z$).
Bei Ankunft -> vacuum_on = False, warte 0.5s.
Sende Retract-Pose. Lösche (pop) den ersten Wert aus master_dropoff_local.
Wechsle zurück zu CHECK_LISTS.

Antworten auf deine Architektur-Fragen
1. Wie schreiben wir Listen, wie sollen wir die am besten übergeben?
In AICA können keine "Listen von Objekten" über die UI-Kabel geschickt werden, sondern nur flache Zahlen-Arrays (signal_type: double_array).
Die Lösung ist das "Flattening" (Flachklopfen) mit einem festen Schrittmaß (Stride):
Ein Frame hat 7 Werte (X, Y, Z, Qx, Qy, Qz, Qw).
Zwei Frames in einer Liste sehen so aus: [X1, Y1, Z1, Qx1, Qy1, Qz1, Qw1, X2, Y2, Z2, Qx2, Qy2, Qz2, Qw2].
Der empfangende Block durchläuft das Array in einer for(int i = 0; i < array.length; i += 7) Schleife, um die Frames wieder zusammenzubauen.
2. Wie published AICA Frames? Wie will es der JTC empfangen?
Der AICA Component Runner abstrahiert ROS völlig. Wenn du in Python eine sr.CartesianPose("pose_name", "base_link") erstellst und mit .set_position([x,y,z]) und .set_orientation([qx,qy,qz,qw]) füllst, bindest du sie an den Output.
Der AICA C++ Kern übersetzt diese CartesianPose unter der Haube automatisch in eine geometry_msgs/msg/PoseStamped und publisht diese auf das ROS Topic, auf dem der JTC lauscht. Du musst in Python keine Header, Timestamps oder Frames zusammenbauen, das erledigt das AICA-Framework anhand des Namens "base_link".
3. Wie kann man Listen von Masken-Eckpunkten weitergeben?
Genau wie bei den Frames. Eine Maske hat 4 Eckpunkte mit jeweils U- und V-Pixeln, also 8 Werte pro Klotz.
YOLO sendet ein double_array: [U1,V1, U2,V2, U3,V3, U4,V4, U1_B,V1_B...].
Der DataHandler iteriert in i += 8 Schritten darüber, greift sich die 8 Werte, berechnet mit klassischer Schulgeometrie den Diagonalschnittpunkt als Center und den Winkel der Kante als Ausrichtung.
